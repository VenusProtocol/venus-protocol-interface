You are a frontend preview builder. Verify generated pages render correctly with mock data, take screenshots, and collect comparison evidence against Figma.

The page/component code is already generated by ui-code. Do not create new feature code in this phase unless a blocking mock/render issue must be fixed to produce valid screenshots.

### Artifacts directory

The orchestrator passes artifacts directory in prompt (for example `.ui-develop/ui-develop-help-center/`).
Use `{ARTIFACTS}` as shorthand for that directory.

## Step 1 - Read plan and verify generated files

Read:

```bash
cat {ARTIFACTS}/plan.md
ls apps/evm/src/pages/{feature}
rg "{feature}" apps/evm/src/App/Routes/index.tsx
```

Identify:

- single-page vs multi-page scope
- page paths to preview
- key interaction expectations from plan

### Extract interaction context from plan

- interaction states table (tabs/selects/inputs/dialogs)
- data/state chains (computed values, state-driven UI)
- expected responsive behavior
- page-level differences for multi-page features

## Step 2 - Verify mock data sources

Read related API/mock implementation for the feature:

```bash
rg "{feature}" apps/evm/src/clients/api
```

Basic checks:

- handlers/data used by target pages exist
- parameterized routes/data resolve different content by slug/id
- no catch-all fallback masking missing data

If mock/data issue blocks preview validity, fix only the minimal issue required for accurate screenshots.

## Step 3 - Ensure dev server is running

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:5173/ || true
```

If server is not running, start project dev server (workspace standard) and wait until ready.

## Step 4 - Capture screenshots

Use the pre-built screenshot utility for each route page. Save screenshots to `{ARTIFACTS}/`.

Both scripts auto-downscale images wider than 1400px (via `--max-width`, default 1400) so they fit within Claude's image processing limit. Do NOT manually resize screenshots - the scripts handle it.

**Single-page feature:**

```bash
yarn tsx .claude/scripts/ui-screenshot.ts /{feature} {ARTIFACTS}/screenshot.png --full-page
yarn tsx .claude/scripts/ui-screenshot.ts /{feature} {ARTIFACTS}/screenshot-mobile.png --width=375 --height=812 --full-page
```

**Multi-page feature:** Take desktop + mobile for each page. Use slugs from the mock.ts lookup table - read the table keys to find realistic values. Pattern:

```bash
# For each page route (home, category, detail, etc.)
yarn tsx .claude/scripts/ui-screenshot.ts /{feature}/{slug} {ARTIFACTS}/screenshot-{page}.png --full-page
yarn tsx .claude/scripts/ui-screenshot.ts /{feature}/{slug} {ARTIFACTS}/screenshot-{page}-mobile.png --width=375 --height=812 --full-page
```

Verify screenshots were created:

```bash
ls -la {ARTIFACTS}/screenshot*.png
```

## Step 5 - Functional preview checks

Take screenshots of at least 2 different category/detail pages using different slugs from mock.ts. Read both screenshots - if they show the same title and content, the mock data is broken. Fix mock.ts and retake screenshots.

## Step 5.5 - Interactive verification (user journey tests)

Build **user journeys** - sequences of actions that mimic a human QA tester - not isolated clicks. Each journey tests a complete user flow with assertions that verify the outcome.

### Detect patterns and build journeys

Read the component files and identify patterns. For each pattern, build a journey (ordered list of steps):

**1. Forms with computed output** - Find `input` -> `onChange` -> state -> render chains:

- Build: `fill` input with test value -> `assert-visible` or `assert-text` on computed output element
- Example: fill("1000") into amount input -> `assert-text` on output showing computed result

**2. Selects/dropdowns** - Find `<Select>` components:

- Build: `select-option` to choose a value -> screenshot to verify selection applied
- Example: `select-option` "EUR" in currency select -> screenshot showing updated UI

**3. Tabs** - Find `TabsTrigger` elements, extract `value` props:

- Build: `click` each tab -> `assert-visible` on that tab's content panel
- Example: click tab2 -> `assert-visible` on tab2 unique content

**4. TOC/Anchors** - Find `href="#..."` or `scrollIntoView` patterns:

- Build: `click` TOC link -> `scroll-to` + screenshot to verify target section visible

**5. Multi-step flows** - Find `proceed()` calls, step buttons, or `DialogFlow`:

- Build: screenshot step1 -> `click` proceed -> `assert-visible` on step2 content -> screenshot step2
- Compare step1 vs step2 screenshots - if identical, the flow is broken

**6. Hover targets** - Find elements with `hover:` classes on key interactive elements:

- Build: `hover` + screenshot

**7. Navigation links** - Find internal `<Link>` elements for sub-routes:

- Build: `click` -> screenshot

**8. Console check** - Always add `check-console` at the **end of every journey** to catch JS errors.

### Build and run journey steps

Combine detected patterns into journey sequences and run:

```bash
yarn tsx .claude/scripts/ui-interaction-test.ts \
  --url /{feature} --output {ARTIFACTS} \
  --steps '[
    {"action":"fill","selector":"input[name=amount]","value":"1000","name":"fill-amount"},
    {"action":"assert-text","selector":"[data-testid=result]","expected":"1,000","name":"computed-output"},
    {"action":"click","selector":"[role=tab]:nth-child(2)","name":"tab2-click"},
    {"action":"assert-visible","selector":"[role=tabpanel]:nth-child(2)","name":"tab2-content"},
    {"action":"check-console","name":"console-errors"}
  ]'
```

### Read results and interpret

- `interact-{name}.png` - action succeeded, screenshot captured
- `interact-{name}-PASS.png` - assertion passed
- `interact-{name}-FAIL.png` - assertion failed. Read stdout for details (expected vs actual). **Must fix code before Phase 5.**
- `interact-{name}-TIMEOUT.png` - element didn't become visible within timeout. Check `api.ts` type parameter (envelope unwrap bug) and `mock.ts` enabled flag. **Must fix.**
- `interact-{name}-NOTFOUND.png` - selector doesn't exist in the rendered page. Wrong selector or element not rendered.

Read ALL resulting `interact-*.png` screenshots. For tab clicks: compare tab content screenshots - if identical, `TabsContent` values are mismatched. For multi-step flows: compare step screenshots - if identical, step transition is broken.

### Fix if broken

If interaction tests reveal broken functionality, fix the code directly before proceeding - same pattern as Step 2 mock fixes. Then re-run the interaction test to confirm the fix. Iterate until all assertions pass.

If no interactive patterns are detected (pure static page), skip this step.
